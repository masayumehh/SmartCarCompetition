项目运行逻辑说明（readmeforqjw）

1. 整体主流程
本车模程序的主入口在 project/user/main.c。
上电后先完成时钟、调试接口、1ms 控制看门狗和底盘/元素模块初始化，然后等待摄像头 mt9v03x 初始化成功。
进入主循环后，每次只在 mt9v03x_finish_flag 置位时处理一帧图像，处理顺序如下：
1）把 DMA 摄像头缓冲复制到 g_img_output，避免处理过程中原始帧继续被覆盖。
2）执行 image_processing，得到二值图和普通巡线中线 g_mid_line。
3）先做发车线/完赛检测。
4）再做环岛、路障、十字、坡道等元素识别，并按优先级决定是否用元素模块输出的 override_mid_line 接管普通中线。
5）把最终选中的中线送入 car_control，输出舵机和电机控制量。
6）如果长时间收不到新帧，则触发掉帧保护，执行 car_emergency_stop。

当前主流程里的元素控制优先级是：
环岛 > 路障 > 十字 > 坡道 > 普通巡线

也就是说，只要更高优先级元素当前激活，下面的普通巡线或低优先级元素就不会接管控制。

2. 图像识别过程
图像处理主逻辑在 project/code/img_processing.c。

2.1 原始图像到二值图
每帧先使用 calculate_otsu_threshold 在图像中下部有效视野内计算 OTSU 自适应阈值。
这样做的目的是尽量避开顶部远端背景、反光和无关区域，让阈值更多根据真正赛道区域来计算。
得到阈值后，程序还会再做一次 tune_binary_threshold 小范围修正，使当前白赛道场地更容易分开黑白区域。
之后通过 binarize_with_threshold 把整幅灰度图转成二值图：白赛道为 255，黑线/黑块为 0。

2.2 上下遮罩
二值化后，会按 tuning_params.h 里的参数把图像最顶部和最底部一小条区域清零。
顶部轻遮罩是为了减小远端噪声和墙面干扰。
底部轻遮罩是为了减小车体附近、极近场反光和赛道脚下的误检。

2.3 中线提取
中线提取使用的是“逐行找左右边界，再取中点”的方式，而不是按列做横向扫描。
具体过程如下：
1）从图像底部往上逐行处理。
2）在每一行里先找左边界 find_left_track_edge 和右边界 find_right_track_edge。
3）如果左右边界都可靠，就取中点作为这一行中线。
4）如果边界不完整，则退回到该行白点重心 get_row_centroid。
5）如果当前行结果与前一行连续轨迹差异过大，则判为不可信，尽量沿用上一行趋势，减少反光和噪点对中线的突然拉扯。
6）所有行处理完后，再对整条中线做一次 smooth_mid_line 平滑。

最终中线数组定义为：
mid_line[y] = 第 y 行赛道中心所在的列号

3. 巡线过程
普通巡线控制逻辑在 project/code/car_control.c。

3.1 中线几何特征提取
car_control 在每一帧都会先调用 analyze_track_features，从中线中提取以下几类量：
1）offset_mean：整条中线相对目标中心 TARGET_LINE 的平均横向偏移。
2）slope：中线整体走势斜率，用最小二乘线性拟合得到。
3）curvature：中线离散二阶差分的平均值，用来近似表示弯曲程度和弯向。
4）valid_points：有效中线点数量。

这里对靠近车前的中下部区域赋予更大权重，所以控制更关注“近场赛道长什么样”，而不是被远端弯道过早带偏。

3.2 赛道有效性判断
程序会统计中下部区域的有效中线点数。
如果有效点过少，则认为当前帧赛道不可靠。
当前策略不是一丢线就急停，而是：
1）短时丢线：先保持上一拍舵机角度和上一拍电机占空比，避免瞬间抽动。
2）连续多帧丢线：触发 failsafe，执行急停和舵机回中。
3）恢复后也不会立刻放开，而是要连续若干帧重新看到有效赛道，才解除保护。

4. 舵机运行逻辑
舵机控制同样在 project/code/car_control.c。

4.1 转向误差组成
舵机不是只看单纯的中线偏移，而是综合以下几项：
1）offset_mean：基础横向误差。
2）slope：中线走势前馈，让车提前顺着弯走。
3）curvature：弯曲程度前馈，让车在大弯中更主动地加舵。

计算出的 steer_error 会先做低通滤波，减少一帧一帧的抖动。

4.2 舵机控制链路
舵机控制链路是：
steer_error -> 转向 PID -> shape_steering_command -> control_steering

其中 shape_steering_command 做了几件关键事情：
1）根据曲率大小提高或降低转向增益。
2）对小舵角做非线性压缩，让直道和小偏差时更柔和。
3）限制进弯、回舵、反向换向时的最大角速度。
4）在反打方向时额外限制切换速度，避免一下子从一边抽到另一边。

最终 control_steering 会把目标角度限制在舵机允许的范围内，再换算成 PWM 占空比输出。

5. 电机运行逻辑
电机控制也在 project/code/car_control.c。

5.1 速度目标怎么来
基础目标速度由上层主流程设定：
1）普通巡线时使用 TUNE_DEFAULT_TARGET_SPEED。
2）进入十字、路障、坡道、环岛等元素时，上层会先 set_target_speed，切到元素专用速度。

在 car_control 内部，这个基础目标速度还会继续被动态减速。
减速依据包括：
1）当前转向误差越大，减速越多。
2）中线斜率越大，减速越多。
3）中线曲率越大，减速越多。
4）有效中线点比例偏低时，再额外减速。

也就是说，程序的思想是：弯越急、线越歪、图像越不稳，车就越慢。

5.2 速度反馈怎么来
两路编码器分别读取左右电机的周期计数，换算成速度反馈后取平均值，作为整车当前速度。

5.3 电机控制链路
电机控制链路是：
目标速度 speed_target -> 速度 PID -> 占空比限幅 -> control_motor_duty

输出阶段还有两层保护：
1）最大占空比不会超过 SPEED_OUTPUT_LIMIT。
2）如果已经决定给动力，但占空比太小又不足以真正起转，就强制抬到 MOTOR_MIN_DUTY_PERCENT。

6. 元素识别过程

6.1 发车线/完赛停车
模块在 project/code/start_line_element.c。
逻辑不是一上电就马上识别发车线，而是：
1）起步后先等待离开发车区。
2）main.c 里还额外设置了 START_LINE_ENABLE_DELAY_MS，目前是 40000ms，也就是运行 40 秒后才重新允许发车线完赛检测。
3）检测方式是在赛道中心附近做纵向扫描，找“黑带 -> 白间隔 -> 黑带”的双黑带结构。
4）再次稳定识别到该结构后，认为小车回到了起点，置位 g_run_completed。
5）主循环看到 g_run_completed 后会持续 car_emergency_stop，锁定停车。

6.2 十字
模块在 project/code/cross_element.c。
十字判断主要看三层宽度关系：
1）下部像正常直道；
2）中部明显变宽；
3）上部又收回到接近正常直道宽度；
4）上下中心差不能太大，避免把普通大弯误识别成十字。

十字激活后不会让车转弯，而是直接生成一条“强制直行”的 override_mid_line，并把速度切到十字专用较低速度。

6.3 路障
模块在 project/code/obstacle_element.c。
路障识别会先建立每一行赛道左右边界，然后只在赛道内部搜索连续黑块。
候选黑块需要满足：
1）宽度、高度、面积合理；
2）左右两侧仍有足够赛道空隙；
3）中心不能太接近赛道中心，否则不好判断从哪侧绕行；
4）在连续多行里位置和宽度不能跳得太厉害，否则更像噪声或别的元素。

识别成功后状态机会经历：
NONE -> APPROACHING -> AVOIDING -> EXITING

绕行控制的核心不是突然把中线猛拉出去，而是按障碍接近程度和当前状态逐渐把中线拉向自由通道，同时把目标速度降下来。

6.4 坡道
模块在 project/code/slope_element.c。
坡道识别本质上是看“赛道宽度是否形成居中、对称、逐渐收窄的梯形”。
状态机会经历：
NONE -> APPROACHING_UP -> CLIMBING -> PLATFORM -> DESCENDING -> EXITING

进入坡道阶段后，会输出更稳定的 guide_center，并按阶段切换不同速度：
1）上坡接近阶段补速；
2）上坡阶段保持动力；
3）平台阶段回收一点速度；
4）下坡阶段进一步保守；
5）出坡阶段平滑释放回普通巡线。

6.5 环岛
模块在 project/code/roundabout_element.c。
环岛识别会先建立“稳定边”和“异常边”，再尝试找 A/B/C 三个关键点来组织补线。
如果识别成功，状态机会经历：
NONE -> ENTERING -> INSIDE -> EXITING

当前 tuning_params.h 中 TUNE_ROUNDABOUT_ENABLE 为 0，说明这版参数默认先关闭了环岛识别，重点放在普通弯道、十字、路障和坡道上。

7. 当前代码里的已知问题

7.1 舵机出弯反打方向还是偏急
虽然 car_control.c 里的 shape_steering_command 已经对“反向换向”和“回舵阶段”做了限速与平滑，但从当前实车表现看，出弯后的反打仍然偏急。
这通常说明以下几类量仍可能偏大：
1）曲率前馈过强；
2）回舵释放速度仍偏快；
3）近场中线在出弯瞬间跳变较大。

7.2 小车运行帧率可能偏低
当前每帧都要执行：
1）整帧拷贝；
2）OTSU 阈值计算；
3）整图二值化；
4）逐行中线提取；
5）发车线、十字、路障、坡道、环岛等多个模块判断；
6）底盘控制和保护逻辑。

因此当场地复杂、元素模块全开、或者后续重新启用图传/调试输出时，帧率有可能下降，进而影响控制周期稳定性。

7.3 环岛当前处于默认关闭状态
虽然环岛代码已经保留，但本次参数里默认关闭了环岛识别。
如果后续赛道包含环岛，需要重新开启并重新调参。

7.4 完赛检测启用时间较晚
当前发车线再次识别被延后到起步后 40 秒才允许启用。
如果实际赛道单圈时间远低于这个值，就会出现“明明回到起点，但不会自动判停”的情况。

8. 后续维护建议
1）优先从 main.c 看总流程，再顺着 image_processing -> 元素模块 -> car_control 的顺序读代码。
2）调车时先保证普通巡线稳定，再逐个打开十字、路障、坡道、环岛。
3）如果继续优化出弯反打，建议先看 car_control.c 里的 STEER_RELEASE_RATE_DPS、STEER_REVERSE_RATE_DPS、STEER_CURVATURE_GAIN 和中线近场权重。
4）如果继续优化帧率，建议优先检查图传频率、OTSU 统计区域大小、元素模块扫描范围，以及是否有不必要的调试输出。
